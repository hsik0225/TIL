# GitHub Actions 용어 설명

## 액션
단계적으로 조합하여 일(Job)을 작성하기 위한 각각의 태스크입니다. 액션은, Workflow의 최소한의 포터블한 구성요소입니다.  GitHub 커뮤니티에서 공유되는 액션을 사용하여 독자적으로 액션을 작성하는 것도, Public한 액션을 커스터마이즈하는 것도 가능하다. Workflow에서 액션을 사용하기 위해서는 그것을 단계적으로 포함해야만 한다.

## 성과물(Artifact)

성과물이란, 코드를 빌드하여 테스트할 때 작성하는 파일입니다. 예를 들어, 성과물에서는 Binary 또는 패키지 파일, 테스트 결과, 로그 파일 등이 있습니다. 성과물은, 그것을 작성한 Workflow의 실행에 연관되고, 다른 일(Job)을 사용하거나, 배ㅐ포하거나 합니다.

## 연속적 인티그레이션(CI)
빈번한 작은 코드 변경을 공유 Repository에 커밋하는 소프트 웨어 개발의 Practice입니다.  GitHub Actions를 사용하면, 코드를 자동적으로 빌드하여 테스트하는 커스텀의 CI Workflow를 작성할 수 있습니다. Repository에서 코드 변경의 Status와 Workflow 안의 각 액션의 상세한 로그를 볼 수 있습니다. CI는 코드 변경에 대해 즉시 피드백을 제공하고, 바로 버그를 검출하고 해결함으로써, 개발자의 시간을 절약합니다.

## 연속적 디플로이먼트(CD)
연속적 디플로이먼트는 연속적 인티그레이션의 위에 구축되어 집니다. 새로운 코드가 커밋되고 CI테스트를 패스한다면, 그 테스트는 자동적으로 생산환경에 디플로이됩니다. GitHub Actions를 사용하면 코드를 임의의 클라우드, 자기 호스트의 서비스나 플랫폼에 Repository에서 자동적으로 디플로이를 하는 커스텀의 CD Workflow를 작성할 수 있습니다. CD는 디플로이먼트의 프로세스를 자동화하고, 테스트뿐인 안정적인 코드 변경을 고객에게 보다 빠르게 디폴리아 하는것으로, 개발자의 시간을 절역합니다.

## 이벤트
Workflow의 실행을 트리거하는 특정 액티비티. 예를 들면, 누군가가 커밋을 Repository에 푸쉬했을 경우, 또는 Issue 혹은 PR이 작성되었을 경우, GitHub에서 액티비티를 발생시키는 것이 가능합니다. 또, Repository의 디스패치Webhook을 사용하여 외부 이벤트가 발생했을 때, Workflow가 실행되게 설정하느 것이 가능합니다.

## GitHub호스트 러너
GitHub는 Linux, Windows, macOS의 러너를 호스트합니다. 잡(Job)은 일반적으로 사용되는 프리인스톨의 소프트웨어를 포함한 가상머신의 새로운 인스턴스에서 실행됩니다. GitHub는 GitHub호스트 러너의 모든 업그레이드나 유지 보수를 합니다. GitHub호스트 러너의 하드웨어 구성은 커스터마이즈할 수 없습니다.

## 잡(Job)
같은 러너 위에서 실행되는 스텝의 조합. 잡의 실행과 관련된 의존계의 룰을 Workflow파일 안에서 정의합니다. 잡은 동시에 병렬적으로 실행하는 것도, 선행하는 잡의 Status에 응하여 순차적으로 실행하는 것도 가능합니다. 예를 들어, Workflow에 코드의 빌드와 테스터라고 하는 2개의 순차적인 잡을 가진 테스트 잡을 빌드 잡의 테스트에 의존시키는 것도 가능합니다. 빌드 잡이 실패했을 경우, 테스트 잡은 실행되지 않습니다. GitHub 호스트 러너는 Workflow 안의 각 잡은 가상환경의 새로운 인스턴스 안에서 실행됩니다.

## 러너
GitHub Actions러너 어플리케이션이 인스톨된 머신입니다. GitHub가 호스트하는 러너를 사용하는 것도, 스스로 러너를 호스트하는 것도 가능합니다. 러너는 실행가능한 잡을 기다립니다. 러너는 잡을 선택하면 그 잡의 액션을 실행하고, 진행상황, 로그, 최종결과의 보고를 GitHub의 반환합니다. 러너는 한 번에 1개의 잡을 실행합니다. 

## 셀프 호스트 러너
러너 어플리케이션이 인스톨된, 당신이 관리하고 유지보수하는 머신입니다. 셀프 호스트 러너는 하드웨어, 운영체제, 소프트웨어 툴에 대해 GItHub 호스트 러너보다 더 컨트롤 할 수 있습니다. 셀프 호스트 러너에서는 대규모적인 잡을 실행하기 때문에 처리능력이나 메모리를 강화한 커스텀 하드웨어 구성을 작성하거나, 로컬 네트워크에서 이용할 수 있는 소프트 웨어를 인스톨하거나, GitHub 호스트 러너에서는 제공되지 않는 운영체제를 선택하는 것이 가능합니다.

## 스텝
스텝은 Commnd나 액션을 실행할 수 있는 개별적인 태스크입니다. 잡에서는 1개 이상의 스텝이 설정됩니다. 잡 안에서 각 스텝은 같은러너 위에서 실행되어 파일 시스템을 사용하여, 그 잡의 복수의 액션에서 정보를 공유하는 것이 가능합니다.

## 가상환경
GitHub호스트 러너의 가상환경에는 그 가상머신의 하드웨어 구성, 운영체제, 인스톨 뿐인 소프트웨어가 포함됩니다.

## Workflow
GitHub에서 임의의 프로젝트를 빌드, 테스트, 패키지, 릴리즈, 또는 배포하기 위해 Repository에 셋업할 수 있는, 설정 가능한 자동 프로세스입니다. Workflow는 한 가지 이상의 잡에서 구성되어, 스케줄하거나, 이벤트에 응해 동작하거나 합니다.

## Workflow File
최소 1개의 잡과 맞춰 Workflow의 설정을 정의하는 YAML파일. 이 파일은 GitHub Repository의 .github/workflows 디렉토리 직하에 있습니다.

## Workflow Run
사전 설정된 이벤트가 발생했을 때 실행되는 Workflow 인스턴스. 각각의 Workflow Run에 대해, 잡, 액션, 로그, Status를 보는 것이 가능하다.
